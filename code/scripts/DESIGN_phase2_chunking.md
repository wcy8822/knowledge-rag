# Loki Phase 2 — 长文档分块向量化 设计方案

> 设计时间: 2026-04-07 | 状态: 待确认

## 一、现状分析

### 当前问题
Phase 1 是"一文一向量"——每个文件生成一条 embedding（最多 3000 字）。
这对短文件还行，但 **54% 的文件超过 5KB**，大文件信息严重丢失：

| 原始文件大小 | 文件数 | 占比 | 当前问题 |
|-------------|--------|------|---------|
| < 1KB | 508 | 16% | ✅ 够用 |
| 1-5KB | 865 | 27% | ⚠️ 略有截断 |
| 5-10KB | 755 | 23% | ❌ 只取了前1/3 |
| 10-30KB | 866 | 27% | ❌ 只取了前1/10 |
| 30-100KB | 130 | 4% | ❌ 严重丢失 |
| 100KB+ | 119 | 4% | ❌ 几乎全丢（含22MB的索引文件） |

### 大文件类型
- **cursor_*.md**: AI对话日志（1-5MB），结构化但冗长
- **2026XXXX-*.md**: Obsidian AI笔记（0.6-3MB），含代码+分析
- **backup_*.sql**: SQL备份（1.4MB），应该跳过
- **外部笔记索引.md**: 22MB索引文件，应该跳过

### 库规模
- 当前: 5407 条 embedding（3个collection）
- 原始文件: 3243 个，总计 193.9 MB
- 预估分块后: **1.5-2万条**（约3-4x增长）

## 二、分块策略

### 核心原则
1. **短文件不动** — < 2KB 的文件不分块，Phase 1 的摘要已够用
2. **语义分块** — 按标题/段落边界切，不是死板按字数切
3. **重叠窗口** — 相邻块有 10-15% 重叠，防止语义断裂
4. **黑名单过滤** — 跳过超大垃圾文件（AI对话日志、SQL备份等）

### 分块规则

| 文件类型 | 策略 | 块大小目标 | 说明 |
|---------|------|-----------|------|
| .md (< 2KB) | 不分块 | - | Phase 1 摘要足够 |
| .md (2K-30KB) | 按 ## 标题切 | 800-1500字/块 | Obsidian 笔记主力 |
| .md (30KB+) | 按 ## 标题切 + 大段再切 | 800-1500字/块 | 长文档 |
| .md (cursor_*) | **跳过** | - | AI对话日志，噪声太大 |
| .pdf | 按页切 | 1页=1块 | pdfminer 按页提取 |
| .docx | 按段落切 | 800-1500字/块 | 合并小段落 |
| .sql | 按语句切 | 1 CREATE/1块 | DDL已有专门collection |
| .xlsx | 不分块 | - | Phase 1 表头提取足够 |
| .txt | 按空行切 | 800-1500字/块 | 类似md处理 |

### 分块参数
```python
CHUNK_MIN = 200      # 块最小字数（太短的合并到上一块）
CHUNK_TARGET = 1000  # 目标字数
CHUNK_MAX = 1500     # 块最大字数（超过则强制切）
OVERLAP = 150        # 重叠窗口字数
```

### 跳过黑名单
```python
SKIP_PATTERNS = [
    'cursor_*.md',           # AI对话日志
    'backup_*.sql',          # SQL备份
    '外部笔记索引.md',        # 22MB索引
    '*node_modules*',
    '*__pycache__*',
]
SKIP_SIZE = 5_000_000  # 5MB以上文件强制跳过
```

## 三、Chunk ID 设计

```
原文件ID:  sha256("path:mtime")[:32]
分块ID:    {原文件ID}_chunk_{序号:03d}

示例:
  a1b2c3d4...._chunk_000  (第1块)
  a1b2c3d4...._chunk_001  (第2块)
```

好处：
- 按前缀能找到同一文件的所有块
- 文件更新时（mtime变化），旧块自动变成孤儿，可清理
- 序号有序，能还原顺序

## 四、Embedding 策略

每个 chunk 的 embed 文本格式：
```
文件: {filename}
路径: {relative_path}
块: {chunk_idx+1}/{total_chunks}
类型: {ext}
---
{chunk_text}
```

对于叙述类 chunk（.md/.pdf/.docx），**不再调 qwen 摘要**：
- Phase 1 是因为整篇太长需要摘要压缩
- Phase 2 每个 chunk 只有 800-1500 字，直接 embed 原文质量更高
- 省掉 qwen 调用，**速度提升 10x+**（从~6s/条 → <1s/条）

## 五、与 Phase 1 的关系

- Phase 2 **不删除** Phase 1 的数据
- Phase 2 在同一个 collection 中追加 chunk 记录
- 搜索时，Phase 1 的"文件级摘要"提供全局视角，Phase 2 的"块级原文"提供精准定位
- 两者互补，不冲突
